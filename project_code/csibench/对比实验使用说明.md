# 对比实验说明文档

## 1. 当前目的

本说明文档用于指导后续 **对比实验（baseline experiments）** 的补充与运行。
当前已经完成了部分模型在 **CSI-Bench** 数据集上的对比实验，后续还需要继续补充更多 baseline 模型，并在多个数据集上复用同一套训练与评估流程。

本说明文档的目标是：

- 统一后续 baseline 模型的添加方式
- 避免重复修改训练流程，减少因改动不一致导致的实验误差
- 保证后续在不同数据集上的对比实验都采用统一 protocol

------







## 2. 当前涉及的主要文件

### 2.1 主模型文件

```text
wavemamba.py
```

用途：

- 存放当前主模型 **WaveMamba** 的正式实现
- 当前对比实验中，会把它作为主方法之一参与比较

当前主要使用的两个版本：

- `wavemamba_default`
- `wavemamba_interp_1x1`

------

### 2.2 对比实验模型文件

```text
baseline_models.py
```

用途：

- 存放所有 **baseline 对比模型**
- 当前已经实现：
  - `resnet18`
  - `lstm`

后续新的 baseline 模型也统一加在这个文件中，避免训练脚本里直接堆模型定义。

------

```text
run_baselines_compare.py
```

用途：

- 统一负责 baseline / 主模型候选的训练、验证、测试与结果保存
- 当前支持：
  - 主模型候选：`wavemamba_default`、`wavemamba_interp_1x1`
  - baseline：`resnet18`、`lstm`
- 后续补充新 baseline 时，主要也是修改这个文件中的模型注册部分

## 4. 当前对比实验的统一 protocol

为了保证对比公平，所有模型都应尽量使用同一套训练与评估流程。

### 4.1 数据划分

当前 protocol 包含四类数据划分：

- `train`
- `val`
- `test_id`
- `test_cross_env`

其中：

- `train` / `val` / `test_id` 来自源环境划分
- `test_cross_env` 来自目标环境（跨环境测试）

------

### 4.2 模型选择方式

当前统一采用：

> **用源域 `val macro-F1` 选择 best checkpoint**

也就是说：

- 训练过程中每个 epoch 在 `val` 上评估
- 根据 `val macro-F1` 保存 best 模型
- 不允许用 `test_cross_env` 参与选模

------

### 4.3 EMA

当前正式结果默认使用：

> **EMA（Exponential Moving Average）**

具体逻辑：

- 训练时原模型参数正常更新
- 同时维护 EMA 参数
- best epoch 仍由 **raw model 的 val macro-F1** 决定
- 若开启 EMA，则最终测试时使用该 best epoch 对应的 **EMA checkpoint**

这样做的目的：

- 提高跨环境稳定性
- 减少 seed 波动
- 缓解模型过度贴合源域的问题

------

### 4.4 保存结果

训练完成后，会在指定保存目录中输出三个文件：

- `json`：保存所有详细实验结果（每个 seed、每个模型、详细指标）
- `csv`：便于快速查看与表格整理
- `md`：适合直接阅读或整理进实验记录

------







## 5. 当前结果文件样式

输出结果汇总表的格式如下：

| Model    | Params(M)        | Best Val F1     | ID Acc          | ID F1           | ID Fall R       | ID Nonfall R    | CrossEnv Acc    | CrossEnv F1     | CrossEnv Fall R | CrossEnv Nonfall R | Best Epoch    |
| -------- | ---------------- | --------------- | --------------- | --------------- | --------------- | --------------- | --------------- | --------------- | --------------- | ------------------ | ------------- |
| resnet18 | 11.1713 ± 0.0000 | 0.9835 ± 0.0015 | 0.9777 ± 0.0016 | 0.9764 ± 0.0017 | 0.9877 ± 0.0032 | 0.9717 ± 0.0026 | 0.6359 ± 0.0284 | 0.6319 ± 0.0286 | 0.5797 ± 0.0248 | 0.7348 ± 0.0382    | 89.40 ± 16.33 |
| lstm     | 0.7665 ± 0.0000  | 0.8862 ± 0.0040 | 0.8755 ± 0.0069 | 0.8675 ± 0.0078 | 0.8387 ± 0.0184 | 0.8977 ± 0.0038 | 0.4549 ± 0.0117 | 0.3950 ± 0.0132 | 0.6015 ± 0.0298 | 0.1964 ± 0.0388    | 76.60 ± 6.97  |

字段含义：

- `Params(M)`：参数量（百万）
- `Best Val F1`：验证集 best macro-F1
- `ID Acc / ID F1`：同分布测试结果
- `CrossEnv Acc / CrossEnv F1`：跨环境测试结果
- `Fall R / Nonfall R`：两类召回率
- `Best Epoch`：被选中的最佳 epoch

## 6. 当前对比实验的简单工作流程

后续任何一个 baseline 模型的加入，基本都遵循以下步骤。

------

### 步骤 1：将模型类加入 `baseline_models.py`

需要做的事：

- 在 `baseline_models.py` 中增加对应模型类
- 保证模型输入与当前数据格式兼容
- 当前输入通常为：

```python
[B, 1, 500, 232]
```

所以模型要么直接接收这个输入，要么在 `forward()` 内部自行转换格式。

例如：

- CNN 类模型：通常直接吃 `[B, 1, 500, 232]`
- RNN/LSTM/GRU 类模型：通常在 `forward()` 中先转成 `[B, 500, 232]`

------

### 步骤 2：在 `run_baselines_compare.py` 中注册新模型

在当前代码中，新增一个 baseline 模型后，必须修改以下 **3 个位置**：

#### ① 在文件顶部导入模型类

例如新增 `GRUBaseline` 后，需要把：

```
from baseline_models import ResNet18Baseline, BiLSTMBaseline
```

改成：

```
from baseline_models import ResNet18Baseline, BiLSTMBaseline, GRUBaseline
```

------

#### ② 在 `get_model_config(model_name)` 中增加该模型的配置项

例如：

```
if model_name == "gru":
    return {
        "type": "gru",
        "num_classes": 2,
        "input_size": 232,
        "hidden_size": 128,
        "num_layers": 2,
        "bidirectional": True,
        "dropout": 0.2,
    }
```

------

#### ③ 在 `build_model(model_name, device)` 中增加实例化逻辑

例如：

```
elif model_type == "gru":
    model = GRUBaseline(**cfg).to(device)
```

------

#### 示例：新增 `gru` 时，必须同时完成以下三步

- 在 `baseline_models.py` 中写好 `GRUBaseline`
- 在 `run_baselines_compare.py` 顶部导入 `GRUBaseline`
- 在 `get_model_config()` 中添加 `"gru"` 的配置
- 在 `build_model()` 中添加 `"gru"` 的实例化逻辑

如果只改其中一部分，脚本会报错，模型不会真正参与实验。

------

### 步骤 3：在 `main()` 中启用该模型

当前脚本实际会运行哪些模型，取决于 `--models` 参数。

有两种启用方式：

#### 方式 A：修改默认模型列表

在 `main()` 中把模型名加入默认值，例如：

```
parser.add_argument(
    "--models",
    type=str,
    nargs="+",
    default=["wavemamba_default", "wavemamba_interp_1x1", "resnet18", "lstm", "gru"]
)
```

这样以后直接运行脚本时，`gru` 也会自动参与实验。

------

#### 方式 B：在命令行中显式指定（推荐）

例如只跑新增的 `gru`：

```
python run_baselines_compare.py --use_ema --eval_mode both --models gru --save_dir ./outputs_compare_add_gru
```

例如同时跑 `gru` 和 `tcn`：

```
python run_baselines_compare.py --use_ema --eval_mode both --models gru tcn --save_dir ./outputs_compare_add_gru_tcn
```

这种方式更推荐，因为更灵活，也不容易误跑太多模型。

------

#### 注意事项

1. `--models` 中写的名字，必须与 `get_model_config()` 中的 `model_name` 完全一致

2. 每次新增模型重新跑实验时，都要修改 `--save_dir`，避免覆盖以前的实验结果

3. 如果运行后提示 `Unknown model_name` 或 `Unknown model type`，优先检查步骤 2 是否改完整

4. > **建议新增模型后，先用 1 个 seed 做一次冒烟测试，确认代码、输入维度、保存路径、结果输出都正常，再正式跑 5 seeds。**

   例如可以举个命令：

   ```
   python run_baselines_compare.py \
     --use_ema \
     --eval_mode both \
     --models gru \
     --seeds 42 \
     --save_dir ./outputs_compare_smoke_gru
   ```

   等确认：

   - 能正常训练
   - 能正常输出 json/csv/md
   - 指标格式没问题

   再跑正式 5 seeds。

------

### 步骤 4：修改保存目录名称

这一点非常重要。

每次加新模型重新跑实验时：

> **一定要修改保存结果的文件夹名称**

否则会覆盖以前已经完成的实验结果。

建议保存目录按实验内容命名，例如：

- `outputs_compare_csibench_v1`
- `outputs_compare_csibench_add_gru`
- `outputs_compare_csibench_add_tcn`
- `outputs_compare_csibench_final`

这样后续回看结果会清晰很多。

------

### 步骤 5：训练结束后检查输出文件

每次实验结束后，需要检查保存目录下是否正确生成：

- `compare_results.json`
- `compare_summary.csv`
- `compare_summary.md`

其中：

- `json` 负责详细留档
- `csv / md` 方便人工查看和整理论文表格

------





## 7. 添加 baseline 时的注意事项

### 7.1 尽量不要改 dataset_loader

baseline 实验的核心原则之一是：

> **只换模型，不改数据处理与训练 protocol**

所以通常不建议为了某个 baseline 单独改 dataset_loader。
如果某个模型需要特殊输入格式，优先在模型内部 `forward()` 中处理，而不是改数据加载逻辑。

------

### 7.2 尽量不要单独给 baseline 特殊训练技巧

为了保证公平对比，baseline 应尽量与主模型共享：

- 相同 split
- 相同优化器
- 相同 epoch 数
- 相同 batch size
- 相同 lr / warmup / weight decay
- 相同 EMA 设置
- 相同 best checkpoint 规则

除非确实有非常必要且合理的原因，否则不要给某个 baseline 单独配特殊技巧。

------

### 7.3 baseline 的目标是“合理、标准、可复现”

baseline 不是越复杂越好。

应优先选择：

- 社区常见
- 结构清晰
- 容易解释
- 容易复现

不建议为了抬高对手而把 baseline 写成一个复杂混合模型，这样会使实验意义变得不清楚。

------

## 8. 当前已完成的 baseline 模型

### 8.1 ResNet18

定位：

- 强 CNN baseline
- 代表经典二维卷积建模方法

特点：

- 对 CSI 这种二维时频输入很自然
- 在当前实验中表现很强
- 是后续主模型最重要的外部对手之一

------

### 8.2 LSTM

定位：

- 经典时序模型 baseline
- 代表“将 CSI 视作时间序列”这一思路

特点：

- 输入通常转成 `[B, T, F]`
- 当前结果明显弱于 ResNet18 和主模型
- 主要用于说明“纯循环时序模型并不适合当前任务”

------







## 9. 后续计划补充的 baseline 模型

下面这些模型是后续推荐继续加入的 baseline。

------

### 9.1 TCN

推荐优先级：**高**

为什么值得对比：

- 代表经典 **时序卷积模型**
- 与 LSTM/GRU 形成互补
- 适合长序列建模
- 能回答：相比传统时序卷积，WaveMamba 是否有优势

作用：

- 补全 baseline 谱系
- 让对比不仅有 CNN / RNN，也有时序卷积路线

------

### 9.2 GRU

推荐优先级：**高**

为什么值得对比：

- 与 LSTM 同属经典循环模型
- 参数量通常更小
- 训练稳定性可能略好于 LSTM
- 能回答：是不是只是 LSTM 不行，换成更轻的 GRU 就能追上来？

作用：

- 验证“经典循环模型整体上是否适合当前任务”

------

### 9.3 ConvNeXt

推荐优先级：**中高**

为什么值得对比：

- ResNet18 已经证明 CNN 很强
- ConvNeXt 可以代表 **更现代的纯 CNN baseline**
- 如果主模型仍然有竞争力，会更有说服力

作用：

- 回答主模型面对现代 CNN 时是否仍然有优势或竞争力

注意：

- ConvNeXt 比 ResNet18 更现代，但实现和训练成本也更高一些
- 建议在完成 TCN / GRU 后再考虑加入

------





## 10. 当前推荐的后续执行顺序

如果继续推进 baseline 对比实验，建议顺序如下：

### 第一优先级

- TCN
- GRU

原因：

- 能快速补齐时序模型谱系
- 训练与实现难度相对可控
- 对论文解释帮助很大

### 第二优先级

- ConvNeXt

原因：

- 可以检验主模型面对更现代 CNN 的表现
- 更偏向增强论文说服力

------

如果最终希望形成一组较完整、较有说服力的 baseline 对照表，建议包含：

- `wavemamba_default`
- `wavemamba_interp_1x1`
- `resnet18`

## 12. 简单总结

后续 baseline 实验的核心思想是：

> **保持数据与训练 protocol 不变，只替换模型。**

具体执行时：

1. 在 `baseline_models.py` 中加入模型类
2. 在 `run_baselines_compare.py` 中注册模型
3. 运行时指定模型列表
4. 修改保存目录，避免覆盖旧实验
5. 检查输出结果文件是否完整

当前已经完成：

- ResNet18
- LSTM

后续优先建议继续加入：

- **TCN**
- **GRU**
- 以及后续的 **ConvNeXt**
